# 读懂一个陌生代码库 (Understand an Unfamiliar Codebase)

这套方法回答一个问题：**面对一个看不懂的项目，按什么顺序去看、去理解源码，才最省力。**

## 一句话方法

> **先看地图，再沿依赖关系反向、自底向上地读——从叶子到入口。**

本质是**按拓扑排序读代码**：先读不依赖任何其他模块的代码，再读依赖它的代码，层层向上，直到最终入口。这样保证**每个模块被读到时，它依赖的一切你都已经理解了**——你永远不会遇到"这个东西调用了一个还没见过的模块"。

## 为什么是自底向上（先理解 why）

一个项目里有两个方向相反的视角：

1. **运行时数据流 = 自顶向下**：用户点一下 → 入口（如 `app.py`）→ agent → 服务 → 最底层 utils，再逐层返回。
2. **依赖关系 = 自底向上**：utils 谁都不依赖；服务依赖 utils；agent 依赖服务；入口依赖一切。

读代码时**反着运行方向走**（自底向上），就能让每个模块的依赖都"已经见过"。这也是讲课 / 搭建用这个顺序的原因：每写、每读一个文件，它需要的东西都已就位，没有"占位""待会再说"，认知负担最小。而**第 0 步的架构图**解决的是"我为什么按这个顺序看"——没有全局地图，你在中途就会迷失，不知道当前模块在整体里扮演什么角色。

## 当本技能触发时，这样做

带着用户针对**他手上这个真实项目**操作，而不是只讲道理。

### 第 0 步——看地图，并画出依赖图

1. **它干什么**：读 README / 文档，弄清项目为用户解决什么问题、怎么跑起来。
2. **目录与入口**：列目录（`tree -L 2`），用文件名先猜职责；找到入口（`main`、`app.py`、`index.js`、CLI、server 路由）。
3. **追 import / 调用关系**：从入口出发，画出"谁依赖谁"。很多 IDE / 工具能自动生成依赖图。
4. **产出一张清晰、美观的 HTML 架构图**（关键产出——不要套固定模板，针对当前项目现场设计）。它必须让人一眼看清四件事：
   - **有哪些工具 / 模块**：每个文件一个分组框，框里用代码风格的小卡片列出函数（`def xxx`）。
   - **有哪些类（class）**：类名标在分组框上（如 `class RagSummarizeService` + `rag_service.py`）。
   - **每个类的方法与属性**：方法写成 `def xxx`、属性写成 `self.xxx`，**分区 / 分列**展示。
   - **依赖关系**：用**箭头连线**表示谁调用 / 依赖谁。
   视觉要求（向"代码结构讲解图"的清晰度看齐）：浅色干净背景；分组框细边、圆角；函数 / 属性做成等宽字体小卡片；可在**左侧放项目文件树**导航；入口 `app.py` 与核心 agent 用**高亮色（如橙色）**突出；连线清爽不打结。技术上用单个**自包含 `.html`**（内联 CSS）+ 一层绝对定位的 SVG（`getBoundingClientRect` 取锚点自动画线，别手算坐标）。
   完成后**打开它给用户看**：Cowork 用 `present_files`；本地用 `open`（macOS）/ `start`（Windows）/ `xdg-open`（Linux）。

### 第 1 步——在图上找到"叶子节点"

那些**不 import 项目内任何其他模块**的文件（`utils`、纯配置、提示词等），就是阅读的起点。

### 第 2 步——沿依赖方向反向，自底向上逐个读

从叶子往根走，每次只读一个模块，确保它调用的东西你都已经看过。这会自然走出**三层**：

- **第一层 · 无依赖的基础设施**：`utils` 这类纯工具，不 import 项目内其他模块，被后面所有人复用——先读它。
- **第二层 · 核心服务**：沿依赖链往上，例如 `vector_store`（依赖 utils）→ `factory` → `rag_service`（依赖 vector_store + factory）→ `agent_tools`（依赖 rag_service）。每一步都只依赖前面读过的东西。
- **第三层 · 组装**：`middleware`、`react_agent` 把服务和工具组装成可运行的 agent。

> 可选（大型项目）：如果叶子太多、不知道先读哪几个，先从入口顺着**一个真实用户操作**追一条调用链，用它筛出"哪些叶子值得先读"，再回到自底向上。

### 第 3 步——最后读入口和胶水代码

`main` / `app.py` / `middleware` 这类把一切串起来的代码放最后——它们依赖所有东西，是整个调用链的顶端。

### 读完自测

用这个检验是否真懂："**要加功能 X，我得改哪些文件、按什么顺序改？**" 能顺畅答出来就说明读通了；答不上来，回到对应那层再读。

## 反过来——从零搭一个新项目

同一套顺序正过来用，就是高效的**写代码顺序**：先画架构图 → 按依赖拓扑排序、**叶子优先**写起 → 每个模块只依赖已写好的模块（不用 mock、写完即可测）→ **入口 / UI 最后写**。

## 把那节课对照一遍（这就是一次拓扑排序）

讲授顺序正是依赖图从叶子到根的拓扑排序，对应上面的三层——每一行只依赖它上面的行：

| 课程顺序 | 文件 / 模块 | 层 | 依赖谁 |
|---|---|---|---|
| 0 | 思维导图 / 架构图 | 看地图 | — |
| 1 | `utils/`（config_handler、path_tools、file_handler、logger_handler、prompt_loader） | 第一层 · 基础设施 | 无 |
| 2 | `vector_store.py`（VectorStoreService）+ `factory.py` | 第二层 · 核心服务 | utils |
| 3 | `rag_service.py`（RagSummarizeService） | 第二层 · 核心服务 | 向量服务、工厂、prompts |
| 4 | `agent_tools.py` | 第二层 · 核心服务 | rag_service 等 |
| 5 | `middleware.py` + `react_agent.py`（ReactAgent） | 第三层 · 组装 | 工具、模型工厂 |
| 6 | `app.py`（Streamlit） | 第三层 · 入口 | 以上全部 |

运行时数据流则**完全反过来**：`app.py` → `ReactAgent` → 工具 → `rag_service` → `vector_store` / `factory` / `utils`，再逐层返回。

需要看这套方法在这个项目上的完整拆解，读 `references/worked-example.md`。

## 几条好用的启发式

- **先看接口，再看实现**：先扫类的方法名、函数签名（模块对外暴露什么），再钻进函数体。
- **名字就是文档**：用目录名 / 类名 / 函数名先建立假设，再去代码里验证。
- **真正的结构是依赖图，目录只是提示**：文件夹布局会骗人，import / 调用关系才是骨架。
- **底层优先投入**：越底层越稳定、越被复用，时间花在这里回报最高。

## 速查清单（可直接照着做）

- [ ] 读了 README，能一句话说出"这项目干什么、怎么跑起来"
- [ ] 列了目录、找到了入口文件
- [ ] 追 import / 调用关系，**产出并打开了一张清晰美观的 HTML 架构图**（看得清工具 / 类 / 方法 / 属性，箭头标明依赖）
- [ ] 在图上标出了**叶子节点**（不 import 项目内其他模块的文件）
- [ ] 从叶子沿依赖方向反向、**自底向上**逐个读，每读一个，它的依赖都已读过
- [ ] **最后**才读入口 / 胶水代码（`app` / `main` / `middleware`）
- [ ] 能回答"加功能 X 要改哪些文件、按什么顺序改"
