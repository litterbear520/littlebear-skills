> 下面是关于 BOSS直聘 的先验提示,标注了整理日期。当作"可能有效的提示"而非"保证正确的事实"——
> 站点结构会变,**第一次真实抓取时务必用 web-access 的 `/eval` 看一眼实际 DOM 再下手**,
> 发现与此处不符就以实际为准,并回写更新本文件。

## 平台特征

- **强反爬 + 登录墙**:未登录或行为异常时,会弹"安全验证"(滑块/点选),或返回内容残缺的页面。复用用户日常 Chrome 的登录态是前提。
- **岗位详情才有完整信息**:搜索列表页(岗位卡片)只有岗位名、薪资、城市、经验、学历、公司、几个技能标签;**完整的"岗位职责""任职要求"原文要进详情页才有**。
- **动态渲染**:列表和详情都是 JS 渲染,直接 curl 拿不到正文,必须走浏览器 CDP。
- **薪资文本格式**(照抄,不要换算,脚本会解析):
  - `25-45K·14薪`(月薪区间 + 年终月数)
  - `20-40K`(无年终标注)
  - `8-12K·13薪`
  - `300-500元/天`(日薪,实习/兼职常见)
  - `面议`

## 有效模式(2026-06-04 真实运行验证)

### 搜索入口(已验证)
- 桌面端搜索结果 URL:
  `https://www.zhipin.com/web/geek/jobs?query=<URL编码关键词>&city=<城市码>`
- **列表是 JS 渲染,要轮询等待**:打开后 4s 可能还没出结果,需"滚动 + 重试 eval"几轮直到出现 `a[href*=job_detail]`(`scrape_zhipin.py` 已这么做)。每页约 17~30 条。
- **翻页**:URL 加 `&page=2`、`&page=3`... 可翻页,每页约 17~30 条。`scrape_zhipin.py` 最多自动翻 5 页。
- 城市码:`city=100010000` 实测返回的多是**深圳**本地结果(疑似按账号/IP 地理默认,并非严格"全国")。**要精确控制城市,优先用 GUI 选城市**让站点自己生成带完整参数的 URL,别赌死记的码。

### 筛选参数(薪资/经验/学历/精确城市)
- 搜索 URL 支持筛选参数(如 `experience=`、`degree=`、`salary=`),但**取值是 BOSS 的内部编码,会变、各站点版本不一,不要凭记忆猜码**。
- 正确做法:用 web-access 在 GUI 里勾好筛选条件 → 站点自己刷新地址栏 URL → 从真实 URL 里把 `query`/`city` 之外的参数原样搬进 config.json 的 `extra_params`(脚本会拼进每页搜索 URL):
  ```json
  {"keywords": ["..."], "city": "101280600", "extra_params": {"experience": "105,106", "degree": "203"}}
  ```
- 精确城市同理:GUI 选完城市,从地址栏拿真实 `city` 码填进 config,比死记城市码可靠。

### 列表 → 详情(已验证)
- 详情链接:列表里 `a[href*=job_detail]` 的 `href` 即完整详情地址 `https://www.zhipin.com/job_detail/<id>.html`,带会话参数,**直接复用、不要裁剪**。
- **列表页薪资被反爬隐藏**:`.job-salary` 的文本是 `-K`(数字用 CSS 伪元素/字体隐藏)。**薪资只能从详情页取**。

### 详情页选择器(已验证)
- 岗位名:`h1`(可能尾随薪资,用 `/\s+\d.*$/` 去尾)。
- 薪资:`.salary`(真实数字,如 `15-18K`、`20-40K·14薪`)。
- 城市/经验/学历:`.text-city` / `.text-experiece`(注意 BOSS 拼写就是 experiece)/ `.text-degree`。
- **公司**:`.sider-company .company-info` 是本岗位真实公司名;`.sider-company` 整段文本去掉"公司基本信息/查看全部职位/公司名"后,剩下的就是 `融资 人数 行业`(如 `天使轮 20-99人 半导体/芯片`)。
- 职责 + 要求正文:`.job-sec-text`(BOSS 把"岗位职责"和"任职资格/任职要求"放在同一段,按这些小标题切分;`split_jd()` 已实现)。

### 提取建议
- 加分项小标题常见写法:`加分项`、`优先`、`以下条件优先`、`满足以下…优先`。其后常跟"我们提供/福利待遇/工作地点"等非要求内容,要截断(`BENEFIT_MARKERS`)。
- 切不开就整段放 `requirements_raw`,**宁可不切,也不要丢内容或改写**。

## 已知陷阱(已验证)

- **公司名易抓错**:页面右侧"相似职位推荐"也有 `.company-name`(还排在前面)。用通配 `[class*=company-name]` 会抓到推荐公司而非本岗位公司。**必须用 `.sider-company` 作用域内的节点**。
- **列表渲染慢**:固定 `sleep 4s` 经常拿到 0 条,要轮询等待(已修)。抓到 0 不要覆盖已有 `jobs.json`(已修)。
- **密集开 tab 触发风控**:短时间批量 `/new` 易弹验证。两次详情之间留间隔(默认 2.5s),子 Agent 分治也要控节奏。
- **滑块/安全验证无法自动绕过**:`DETAIL_JS` 会检测并置 `sec=true`;遇到就停下,请用户在自己的 Chrome 里手动过验证,完成后重跑(已抓到的不丢)。不要尝试自动破解验证码。
- **"职位不存在/页面不见了"未必是真的**:常是访问方式异常(URL 缺参、被风控)导致。回退到"在列表页点击进入"的 GUI 方式。
- **薪资 `面议`/日薪**:`build_report.py` 解析不了的薪资项会跳过薪资统计但仍计入岗位数,并在报告标注可解析样本量。

## 抓取数量与样本

- 默认每个关键词抓 35 个带完整详情的岗位即进入分析;样本越大词频越稳,但越慢、越容易触发验证。
- 若中途反复触发验证导致抓不满,**拿到多少就用多少**,在报告里如实标注实际样本量,不要为凑数硬刚。
