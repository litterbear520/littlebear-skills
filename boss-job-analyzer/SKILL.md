# BOSS直聘 岗位抓取与分析

把一句口语化的需求(如"看看 boss 上 AI Agent 工程师都要什么")变成:

1. 一批结构化的岗位数据 `jobs.json`(+ 可选的逐岗 Markdown)
2. 一份**单文件、离线可看、Anthropic 博客风**的 HTML 报告 `report.html`:顶部概览数字 + 5 个分析维度(每个维度带"一句话关键结论") + 可点击展开看 BOSS 原文的岗位卡片墙

## 心法

- 抓取层**不要自己造浏览器轮子**,交给 `web-access`(CDP 直连用户日常 Chrome,天然带 BOSS 登录态、内置反爬防护)。
- 分工:**"这个岗位涉及哪些技能/语言/职责"这种理解判断,由你(模型)读 JD 原文来做**;"数数、算占比、算共现"这种必须可复现的算术,交给脚本 `scripts/build_report.py`。
  - 为什么这样分:技能识别用写死的关键词词典(过去的 `TECH_TERMS` 等)**只对它预设的岗位类型(AI/Agent)有效**——一旦用户搜"Java 后端""数据分析师""产品经理",词典要么大面积落空、要么漏掉真正高频的技能,分布直接失真。你读真实 JD 能适配**任意岗位**,还能抓到长尾技能;但占比/共现这类统计必须确定性,所以你只负责"判定每个岗位的标签",脚本负责"按标签计数"。
  - 落地方式:第 2 步你为**每个岗位**输出归一化标签 + 跨岗位洞察,写进 `analysis.json`,第 3 步传给脚本渲染。脚本里仍保留正则词典,但**仅作没有 `analysis.json` 时的降级兜底**(报告脚注会注明降级),正常流程一定要走 Claude 标注。
- 抓取用脚本 `scripts/scrape_zhipin.py`,选择器已真实验证。模型的职责是把流程串起来、分析 JD 规律、处理登录/验证码等需要判断的环节。
- 站点专属知识(搜索 URL、选择器、薪资反爬、城市码、节奏)都在 `references/zhipin-site-notes.md`,**开工前先读它**。
- 岗位关键词联想映射在 `references/keyword-groups.md`,用户给出关键词后查它来扩展搜索覆盖面。

## 依赖:web-access 技能(抓取 + 打开报告都靠它的 CDP)

BOSS直聘 反爬强,本 skill 不自己造浏览器轮子,而是用 [web-access](https://github.com/eze-is/web-access) 的 CDP proxy 复用用户日常 Chrome 的登录态抓取(`scripts/scrape_zhipin.py` 连 `localhost:3456`),报告也用它在用户 Chrome 里打开——**这是硬依赖,没它抓不了**。

web-access 本身跨工具(Claude Code / Cursor / Codex 都能装),所以新用户没装时**先问用户**要不要装,再给方式:
- 方式一(推荐,跨工具一键):`npx skills add eze-is/web-access`
- 方式二(让我代装):用户说「帮我安装这个 skill:https://github.com/eze-is/web-access」

装好后按第 0 步跑 `check-deps.mjs` 确认 proxy ready 再继续。Python 侧零三方依赖(标准库即可),无需 pip 安装。

## 第 0 步:信息收集 + 登录(触发后先做这个)

不要急着抓。先把这几件事问清楚/准备好,缺什么补什么(用 `AskUserQuestion` 或直接提问):

1. **岗位关键词 + 智能扩展**(必需):
   - 用户说了关键词后,**先读 `references/keyword-groups.md`** 查找相关岗位叫法。
   - 把扩展后的关键词列表**展示给用户确认**,例如:
     > "你说的是 Agent 工程师,这些相关岗位也值得一起看:AI Agent开发、大模型应用工程师、LLM应用开发、Harness工程师。要搜哪几个?"
   - 如果映射表没命中,根据岗位名自行联想 2-4 个相关叫法,同样列给用户选。
   - **用户确认后**,选中的关键词全部放进 config.json 的 `keywords` 数组,一次抓取多个。
   - 没说关键词就问"你想看 BOSS 上哪个岗位?"。
2. **城市**:用户说了(如"杭州的")就用对应城市码;**没说城市就问**"看哪个城市的?还是全国?"——不要默认替用户决定。城市码见站点笔记;拿不准就走 GUI 选城市。
3. **数量**:**问"抓多少条?"**,默认 35。越多分析越稳,但越慢、越易触发验证。
4. **可选筛选**(默认不筛、全量抓):用户主动提了再加,如"只看 20K+""只看 3-5 年""只看本科及以上"。落地方式:用 web-access 在 BOSS 搜索页 GUI 里勾上筛选条件,站点会刷新地址栏 URL,把其中 `query`/`city` 之外的参数原样搬进 config.json 的 `extra_params`(取值是 BOSS 内部编码,会变,**不要凭记忆猜码**)。精确城市同理——GUI 选完城市从地址栏拿真实 `city` 码。
5. **浏览器与登录**:
   - 用 `Skill` 工具调用 `web-access`,跑它的 `check-deps.mjs` 确认 CDP proxy ready(Node + Chrome 远程调试)。**没装 web-access?** 按上文「依赖」节先引导用户安装,再继续。**必须**先把 web-access 的反爬风险须知展示给用户。
   - 开 BOSS 搜索页探一下能否拿到岗位内容。**若发现未登录/拿不到内容**,明确请用户登录:
     > "BOSS直聘 当前未登录或拿不到岗位详情,请在你的 Chrome 里登录 BOSS直聘,登录完成后告诉我继续。"
   - 登录后无需重启任何东西,刷新继续即可。

**信息没齐(尤其关键词、城市、数量)不要开抓。**

## 第 1 步:抓取(脚本主路径)

把参数写进一个 **UTF-8 的 config.json**(中文经文件读入,避开 Windows 命令行参数编码问题):

```json
{"keywords": ["AI Agent开发工程师"], "city": "100010000", "count": 35, "out": "jobs.json", "delay": 2.5, "resume": true,
 "extra_params": {"experience": "105", "degree": "203"}}
```

- `resume`(默认 `true`):若 `out` 指向的文件已存在,自动加载已抓岗位并跳过已有岗位,只抓增量。设为 `false` 则全量重抓。
- `extra_params`(可选):原样拼进搜索 URL 的筛选参数(薪资/经验/学历/精确城市),从 GUI 生成的真实 URL 里复制,见第 0 步与站点笔记。
- `max_pages`(默认 5)/`proxy`(默认 `http://localhost:3456`,也可用环境变量 `WEB_ACCESS_PROXY`):一般不用动。

然后(Windows 建议加 `PYTHONUTF8=1`):

```bash
PYTHONUTF8=1 python "<技能目录>/scripts/scrape_zhipin.py" config.json
```

脚本会:进搜索页(轮询等待渲染)→ 取详情链接 → 逐个进详情页抽取「岗位名/真实薪资/城市/经验/学历/公司(行业·规模·融资)/技能标签/福利/HR活跃度/职责·要求·加分项原文」→ 切分 → 写 `jobs.json`。带节奏间隔,抓到 0 不覆盖旧文件。

**翻页抓取**:当第 1 页链接不足 `count` 时,自动翻到第 2、3…页(默认最多 5 页,可配 `max_pages`),直到凑够目标数或无更多结果。**去重按岗位 ID**(`/job_detail/<id>`)而非完整 URL——同一岗位经不同入口的 URL 带不同 `securityId` 会话参数,按 URL 去重会把同一岗位抓成多条、虚增样本。

**断点续抓**:`resume: true` 时,若 `jobs.json` 已存在,自动跳过已抓 URL 只抓增量,中途被风控中断后重跑即可无损续上。

**失败重试**:单个详情页提取失败会自动重试最多 3 次,仍失败才跳过。

**增量保存**:每抓 5 个岗位自动存盘一次,中途崩溃或风控中断也不丢已抓数据。

**连接检查**:脚本启动时先验证 CDP proxy 是否可达,不可达直接报错,避免白跑。

**反爬纪律**:不要把 `delay` 调太小。脚本检测到安全验证(搜索页或详情页)会**自动熔断整个抓取**(不会换关键词硬试——风控是账号级的,硬试只会越触发越严),已抓数据已保存。此时请用户在 Chrome 里手动过验证,完成后重跑即自动续抓。抓不满就用现有的,在报告里如实标注样本量,别硬刚。

**脚本失效时的回退**:若列表恒为 0 或字段恒空,多半是 BOSS 改版导致选择器变化。此时回退到 `web-access` 自适应驱动(用 `/eval` 看真实 DOM、像人一样浏览),按真实结构抓取,并**据实更新 `scrape_zhipin.py` 与站点笔记**。`jobs.json` 的字段约定见下方。

**用一个流程还是多个子代理并行,由你自己判断**:单关键词、量不大时,主流程一把跑完即可;抓多个关键词、或量很大时,可把不同关键词派给**多个**子代理并行(各自在共享 Chrome 里开后台 tab)。但抓取共用同一个登录态浏览器,并发太高会触发风控,**务必限并发(2–3 个)、错峰**——这里不是越多越快。

### jobs.json 字段约定(`build_report.py` 依赖这些字段名)

```json
{
  "keyword": "AI Agent开发工程师",
  "title": "AI Agent 开发工程师",
  "company": "深圳市元芯动力科技",
  "company_info": "天使轮 · 20-99人 · 半导体/芯片",
  "city": "深圳",
  "salary_raw": "25-45K·14薪",
  "experience": "1-3年",
  "education": "本科",
  "tags": ["大语言模型", "Python", "Agent"],
  "benefits": ["五险一金", "带薪年假", "年终奖"],
  "hr_active": "刚刚活跃",
  "responsibilities_raw": "岗位职责原文,逐条保留,不改写不总结",
  "requirements_raw": "任职要求原文,逐条保留",
  "bonus_raw": "加分项/优先项原文(没有就空字符串)",
  "detail_url": "https://www.zhipin.com/job_detail/xxxx.html"
}
```

- `salary_raw`:照抄 BOSS 详情页薪资文本(`20-40K·14薪`/`面议`),**不要换算**,`build_report.py` 会解析。列表页薪资被反爬隐藏成 `-K`,**必须从详情页取**。
- `responsibilities_raw / requirements_raw / bonus_raw`:**原文照搬**,任何总结都会让词频失真。
- `tags`:BOSS 给岗位打的技能标签(`.job-keyword-list`),会并入技术栈识别。`benefits`:福利标签。`hr_active`:HR 活跃度文案。
- `keyword`:这条来自哪个搜索词,多关键词分组靠它。

## 第 2 步:分析 JD 规律 + 标注每岗技能(你来做,核心)

抓取完成后,**你(模型)逐个读取 `jobs.json` 里每个岗位的 `title`、`tags`、`requirements_raw`、`responsibilities_raw`、`bonus_raw`**,产出一个 `analysis.json`,它有两部分:

**① `per_job`:为每个岗位标三类归一化标签**(替代过去的正则词典,这是本技能技能分布的唯一数据源):
- `tech`:该岗位涉及的技术栈/框架/工具/领域概念(规范展示名,如 `LangChain`、`RAG`、`向量数据库`、`Docker/K8s`、`PyTorch`、`微服务`)。
- `langs`:编程语言(`Python`、`Java`、`Go`、`TypeScript`…)。
- `duties`:核心职责动作,≤6 个/岗,取最能代表该岗的(如 `模型微调`、`RAG开发`、`Agent开发`、`系统设计`、`性能优化`)。

**② `insights`:通读全部岗位后提炼 6-10 条跨岗位共性规律**(框架/工具组合、经验模式、新兴技能趋势、技能与薪资关联、不看原文注意不到的隐性模式)。好洞察的标准:有**具体数字**支撑、说出**和直觉不同**的东西(人人都知道"要会 Python"不算洞察)、对求职者**可行动**。

```json
{
  "per_job": [
    {"detail_url": "https://...(照抄 jobs.json 里的,脚本靠它对齐)", "tech": ["LangChain", "RAG"], "langs": ["Python"], "duties": ["Agent开发", "模型微调"]}
  ],
  "insights": [
    {"title": "5-10字标题", "body": "详细描述,含<b>高亮</b>和具体数字", "pct": 覆盖率数字}
  ]
}
```

标注纪律,直接决定统计质量:
- **归一化**:同一概念跨岗位必须用**完全一致的字符串**(`langchain`/`LangChain`/`Lang Chain` 都写成 `LangChain`),否则脚本会当成不同项,占比全乱。
- **只标真实出现或明确要求的**,不要凭岗位名脑补;JD 没提的不标。
- **适配该岗位实际内容**,不要套用固定词表——这正是改用 Claude 分析的意义(对任意岗位都准)。
- `per_job` 应覆盖全部岗位;漏标的岗位在该维度不计入,会拉低样本。`detail_url` 照抄 `jobs.json` 里的即可,脚本按岗位 ID(`/job_detail/<id>`)对齐,带不带会话参数都能对上。

**标注完先自检,干净了再进第 3 步**——漏标和归一化不一致从最终报告里肉眼看不出来,必须用脚本查:

```bash
PYTHONUTF8=1 python "<技能目录>/scripts/check_analysis.py" jobs.json analysis.json
```

它会确定性地报告:漏标岗位清单、对不上号的 URL、疑似同义变体(如 `K8s`/`k8s`)、全空标注、insights 的 pct 合法性。发现问题就修 `analysis.json` 重查,全部通过(退出码 0)再生成报告。

**用几个执行单元由你按岗位数量和上下文自己判断**——不必非得派子代理:岗位少(约 ≤20)时,你直接读完标完最省事,或派**一个**子代理代劳;岗位多时,把岗位分批派给**多个**子代理并行提速。无论一个还是多个,都要统一归一化口径(把规范名清单一并交代给每个子代理),并行回来后再做一遍**合并 + 归一对账**(把 `LangChain`/`langchain`/`Lang Chain` 这种并到一个名),汇成一个 `analysis.json`。

## 第 3 步:生成报告

```bash
PYTHONUTF8=1 python "<技能目录>/scripts/build_report.py" jobs.json --analysis analysis.json --out report.html
```

零三方依赖。`--analysis` 传入你第 2 步的标注:脚本据此**确定性地**计数技术栈/语言/职责的岗位占比、算技能共现,并渲染你的洞察;薪资解析、学历/年限/薪资分布等客观统计照旧由脚本完成。**不传 `--analysis` 会退回内置正则词典(降级,报告脚注会注明),正常流程务必传。**

跑完用 `web-access` 在用户 Chrome 里打开 `report.html`(file:// 即可)给用户看,并口述 3~5 条关键发现。

> **注意**: 完整步骤是 0-1-2-3。第 2 步(读 JD、标注每岗技能 + 提炼洞察)是核心,由你完成而非脚本——脚本只对你的标注做确定性统计。

## 报告设计(report.html 固定形态)

- **风格**:Anthropic / Claude 博客风——奶油纸底 `#F0EEE6`、陶土橙 `#C15F3C` 强调、暖墨正文、克制留白、杂志式分区编号、载入错峰动画。
- **深色模式**:右上角按钮一键切换,深色配色(暗灰底 `#1A1A1F`、暖橙 `#E07850`)自动保存到 localStorage;也跟随系统 `prefers-color-scheme` 自动匹配。
- **字体**:拉丁用 **Inter**、中文用 **Noto Sans SC**(均为 Google 开源字体,经 `fonts.loli.net` 镜像在线加载,国内可达);苹果设备优先 `-apple-system`/`PingFang SC` 原生。**离线时 Web 字体不加载会自动降级到系统字体,版式不破**。要换字体改 `HTML_SHELL` 的 `<link>` 与 `:root` 的 `--sans/--num`。
- **总结分析**:概览下方独立一栏,汇总核心技术栈、主流语言、经验/学历门槛、薪资水平、差异化加分项,顶部有"学习建议" callout——帮用户快速看清该学什么。
- **维度**(①②④ 的技能分布来自你第 2 步的 `analysis.json` 标注,脚本只做计数):① 技术栈/框架(雷达图 + 条形图 + 词云 + 技能共现对)② 编程语言占比(甜甜圈图)③ 学历/年限/薪资分布 + 薪资×经验(脚本客观统计)④ 加分项(卡片网格,按原文聚合) + 高频职责关键词(药丸云)⑤ **市场规律洞察**(你在第 2 步提炼的 6-10 条跨岗位共性规律,如框架组合要求、经验模式、新兴技能趋势等)。没有 `analysis.json` 时 ①②④⑤ 回退正则词典兜底。每个维度顶部有陶土色"一句话关键结论"条,最高项深色高亮。空数据的维度自动跳过、编号顺延。
- **图表交互**:所有条形图和柱状图支持 hover 高亮(条变粗 + 亮度增加),每行带自定义 tooltip(`data-tip`)显示详细数据;动画逐条错开,视觉层次清晰。图表全部为手写 CSS/SVG,**不引入图表库**——单文件离线报告引入 ECharts 这类库会让体积膨胀数倍、canvas 在打印/PDF 导出时易糊,且默认审美与本报告风格冲突。
- **浮动目录**:左侧固定侧栏,点击跳转到各维度,滚动时自动高亮当前位置(宽屏≥1200px 显示)。
- **卡片墙**:每岗一张卡(岗位名/薪资/公司·行业/技能标签/福利标签/HR活跃度 ● 徽标),**点击展开**看该岗位 BOSS 原始职责、要求、加分项 + 详情链接。卡片墙顶部有**搜索框**(按岗位名/公司/技能全文搜索)、**经验/学历筛选**、**薪资排序**(从高到低/从低到高),实时过滤并显示匹配数量。
- **PDF 导出**:右上角按钮触发浏览器打印,自带打印样式(隐藏工具栏/控件,双栏卡片,白底)。
- **回到顶部**:滚动超过 400px 后右下角出现回到顶部按钮。

技能识别已不靠词典——要让某个技能更准地被归类,改进你第 2 步标注的归一化口径即可。`build_report.py` 顶部的 `TECH_TERMS / LANG_TERMS / DUTY_TERMS` 现在只在没有 `analysis.json` 的降级路径里用到。

## 不要做的事

- 不要绕过 BOSS 登录或反爬硬爬——本技能就是复用用户合法登录态、像人一样浏览。
- 不要把模型对岗位的"印象"写进 `*_raw` 字段;只照搬页面原文。
- 不要在抓取阶段做统计。分工是固定的:**技能/语言/职责的"判定"是你第 2 步的事(写进 `analysis.json`),计数/占比/共现是脚本的事**;别让脚本回去猜技能,也别自己在报告里手算占比。
